# 12 自主检索型工具图谱 Agent 详细实施计划

## 12.1 文档目的

这份文档用于约束后续 `foodEpic-agent` 的主线实现方向：

- 不把整理好的视频内容主动塞给大模型。
- 不把系统做成“图片问答器”或“证据 prompt 包装器”。
- 让大模型只拿到问题、工具、工作记忆和上一步返回，自主决定去查图谱、查原始视频、查区域、查 OCR、查音频。

这里的“图谱”不是答案库，而是长期记忆层。

这里的“agent”不是一句大 prompt，而是：

`问题 -> 规划 -> 工具调用 -> 写回长期记忆 -> 再检索 -> 最终回答`

---

## 12.2 最高优先级约束

### 12.2.1 禁止主动灌输整理内容

默认情况下，不允许在问题开始时直接给模型：

- 全量帧描述。
- 全量视频摘要。
- 人工整理好的 cooking 流程。
- 某个题目的证据包。
- 人工选好的关键帧集合。
- 某个视频中所有可见食材、器具、步骤的长文本列表。

默认只允许给模型：

- 问题。
- 选项。
- 工具列表和 schema。
- 当前工作记忆。
- 上一步工具返回。

这意味着：

- 模型自己决定下一步检索什么。
- 模型自己决定什么时候回到原始数据。
- 模型自己决定什么时候把新观察写回图谱。

### 12.2.2 图谱只是外部长期记忆，不替代原始证据

图谱中允许保存：

- 时间片段。
- 帧。
- 区域。
- OCR 读数。
- 音频峰值事件。
- 食材加入事件。
- 状态变化。
- 位置关系。
- 已观察到的 timeline 总结。

但图谱不能变成终点。

每个关键节点都必须能反查到：

- 原始视频。
- 时间点或时间段。
- 相关帧。
- 相关区域图。
- OCR 来源图。
- 音频来源片段。

也就是说，图谱回答不了时，agent 必须有能力重新去看原始证据。

### 12.2.3 图谱中不得写入评测真值

允许写入的内容只能是评测时本来就允许看到的证据及其派生观察：

- 视频。
- 音频。
- 抽帧。
- 局部区域图。
- 自动 OCR。
- 自动视觉观察。
- agent 执行过程中产生的观察结论。

不允许写入：

- benchmark 标准答案。
- 标注真值本身。
- 测试时理论上不可见的信息。

---

## 12.3 最终想要的 agent 能力

最终 agent 不应只是“回答一道题”，而应具备以下能力。

### 12.3.1 面向单题的证据推理能力

给定一个问题，agent 能：

- 识别问题需要什么类型的证据。
- 优先查询已有结构化记忆。
- 在证据不足时回到原始视频重新观察。
- 根据新观察写回图谱。
- 给出答案和对应证据链。

### 12.3.2 面向单视频多题的记忆复用能力

给定同一个视频的一组问题，agent 能：

- 复用前面问题中已经观察过的时间段。
- 复用已经识别出的食材、器具、位置、状态变化。
- 复用 OCR、音频峰值、timeline 事件。
- 避免每一题都从零开始重新抽帧。

### 12.3.3 面向真实应用的开放式查询能力

不仅是四选一 VQA，后续还应支持：

- “这道菜什么时候加了盐？”
- “鸡蛋打进碗里之后还做了什么？”
- “这个称上的数字最后变成了多少？”
- “为什么这一步判断应该选 B 而不是 D？”

如果系统只能做 benchmark 选择题，而不能回查证据和解释过程，那么它仍然不是合格 agent。

---

## 12.4 系统结构

### 12.4.1 Memory Layer

职责：

- 保存跨步骤、跨题目的长期记忆。
- 作为低成本优先检索层。
- 为原始数据回查提供索引。

建议节点：

- `video`
- `segment`
- `frame`
- `region`
- `timeline_event`
- `ingredient_event`
- `state_change`
- `object_state`
- `location_relation`
- `ocr_reading`
- `audio_event`
- `agent_observation`

建议边：

- `contains`
- `before`
- `after`
- `supports`
- `derived_from`
- `co_occurs`
- `same_object`
- `same_ingredient`
- `same_step`
- `refers_to`

节点最小字段：

- `node_id`
- `node_type`
- `video_id`
- `start_time`
- `end_time`
- `label`
- `attributes`
- `evidence_paths`
- `source_tool`
- `confidence`

### 12.4.2 Tool Layer

所有证据访问必须工具化。

工具至少分为五类：

1. 图谱检索工具
2. 原始视频访问工具
3. 图像区域访问工具
4. 感知解析工具
5. 图谱写回工具

### 12.4.3 Planner Layer

planner 只做决策，不做“大段总结后直接回答”。

planner 要解决的问题是：

- 当前问题属于哪类任务。
- 当前缺少什么证据。
- 下一步最该用哪个工具。
- 是否应该先查图谱，还是先回到原始证据。

### 12.4.4 Execution Layer

executor 负责：

- 执行工具。
- 更新工作记忆。
- 自动把关键观察写回长期记忆。
- 控制步数、失败、暂停和恢复。
- 记录 trace。

### 12.4.5 Evaluation Layer

评测层不只是看准确率，还要看：

- 是否真正复用了记忆。
- 是否减少了重复抽帧和重复观察。
- 是否能返回可回查证据。
- 每类题用了哪些工具。
- 成本和时延是否可接受。

---

## 12.5 大模型的职责边界

大模型只负责四件事：

1. 形成当前假设。
2. 决定下一步工具。
3. 判断证据是否足够。
4. 在证据足够时输出答案。

大模型不负责：

1. 不经工具直接理解整段长视频。
2. 靠人整理好的摘要直接答题。
3. 把图谱当作标准答案库。

如果一个答案的核心信息来自“预先整理好的大段文本”，那么这不是我们要的系统。

---

## 12.6 工作记忆设计

建议统一状态：

```json
{
  "video_id": "",
  "task_family": "",
  "question": "",
  "choices": [],
  "hypotheses": [],
  "evidence": [],
  "visited_times": [],
  "visited_nodes": [],
  "artifacts": [],
  "open_questions": [],
  "confidence": 0.0
}
```

补充要求：

- `evidence` 只放当前题真正相关的证据。
- `visited_times` 记录已经观察过的时间点或时间段。
- `visited_nodes` 记录已经命中过的图谱节点。
- `artifacts` 记录抽出的帧、区域图、OCR 结果图等。
- `open_questions` 明确写出当前仍未解决的证据缺口。

## 12.7 必须具备的工具能力

### 12.7.1 图谱检索工具

必须长期保留并持续增强：

- `query_time`
- `query_object`
- `query_event`
- `query_state`
- `query_location`
- `query_region`
- `query_ocr`
- `get_neighbors`

这些工具解决的是：

- 是否已经有可复用的观察。
- 某个时间附近已经写过什么。
- 某个对象或食材之前出现在哪里。
- 某个位置关系、OCR 读数、状态变化是否已被记录。

### 12.7.2 原始视频访问工具

必须具备：

- `extract_frame_at_time`
- `extract_frames_for_range`
- `sample_sparse_frames`
- `sample_frames_around_peaks`
- `extract_input_reference_frames`

设计要求：

- 不要抽连续相邻帧。
- 默认用稀疏抽样。
- 只在候选时间窗口内补帧。
- 对需要比较前后变化的问题，要保留前后多时刻视图。

### 12.7.3 区域与空间访问工具

必须具备：

- `render_bbox_overlay`
- `extract_region_with_context`
- `resolve_bbox_reference`

关键原则：

- 默认优先画框而不是纯裁剪。
- 保留周边上下文，避免丢失空间关系。
- 对涉及指代的题目，要能把“左边那个碗”“靠近锅的瓶子”落到具体区域。

### 12.7.4 感知解析工具

必须具备：

- `inspect_visual_evidence`
- `run_ocr_on_image`
- `run_ocr_on_region`
- `detect_audio_peaks`
- `identify_image_ingredients`
- `infer_visual_mcq`
- `infer_action_intent`
- `infer_action_mechanism`
- `infer_gaze_target_with_context`

关键原则：

- 不做离线全量描述灌输。
- 只在需要时对指定帧、指定区域、指定时间段调用感知工具。
- 新观察必须可写回、可复用，而不是一次性消费。

### 12.7.5 图谱写回工具

必须具备：

- `write_observation`
- `write_frame_observation`
- `write_region_observation`
- `write_ocr_reading`
- `write_audio_event`
- `write_timeline_summary`
- `write_state_change`

写回要求：

- 必须附带时间信息。
- 必须附带证据路径。
- 必须附带来源工具。
- 必须附带置信度。

## 12.8 问题求解主循环

每个问题必须遵循以下主循环。

### Step 1：解析问题

识别：

- 题型。
- 核心对象。
- 涉及时间点还是时间段。
- 是否是状态变化。
- 是否是 OCR/称重类。
- 是否是位置/方位类。
- 是否需要跨时间比较。

### Step 2：形成初始假设

planner 只输出：

- 当前假设。
- 当前证据缺口。
- 下一步工具。
- 为什么先用这个工具。

### Step 3：优先查询长期记忆

默认优先走图谱检索：

- 查时间附近节点。
- 查对象相关节点。
- 查状态、位置、OCR、region 相关节点。
- 查前序问题已写回的 timeline 总结。

### Step 4：判断是否缺原始证据

如果长期记忆不够，必须明确缺的是哪类证据：

- 缺更细的时间证据。
- 缺局部区域证据。
- 缺 OCR 读数。
- 缺动作变化前后对比。
- 缺音频触发的关键时刻。

### Step 5：回查原始数据

根据缺口选择工具：

- 时间定位问题：先稀疏抽帧，再局部补帧。
- 称重/OCR 问题：先区域定位，再 OCR。
- 指代/方位问题：先画框保留上下文，再观察。
- 步骤/流程问题：先音频峰值，再候选时间段补帧，再写 timeline。

### Step 6：写回长期记忆

任何新观察都不应只停留在 prompt 中。

必须写回：

- frame observation
- region observation
- OCR reading
- audio event
- state change
- timeline summary

### Step 7：判断是否足够回答

如果还不能区分选项，继续循环。

如果能区分选项，再输出答案。

## 12.9 与“简单问答系统”的本质区别

这个项目后续必须明确和以下系统区分开：

### 12.9.1 区别于 text-only 猜题

text-only 只能靠先验和数据偏置猜。

我们的系统应依赖：

- 视频原始证据。
- OCR 证据。
- 区域证据。
- 时间线证据。
- 图谱中的已积累观察。

### 12.9.2 区别于“给图片再问一句”的 VLM

单次 VLM QA 的问题是：

- 没有长期记忆。
- 不能跨问题复用。
- 不能主动选帧。
- 不能主动细化区域。
- 不能形成 timeline。

我们的系统要比它多出：

- 规划能力。
- 工具调用能力。
- 长期记忆能力。
- 原始证据回查能力。

### 12.9.3 区别于“整理证据后再喂模型”

如果我们先把答案相关信息全整理好再给模型，那么真正工作的不是 agent，而是人工 pipeline。

我们要验证的是：

- 模型会不会自己决定查什么。
- 图谱能不能减少重复观察。
- 多次工具调用能不能形成真正证据链。

## 12.10 分阶段实施计划

### Phase A：固化当前 agent 约束

目标：

- 把“只给问题和工具，不给整理内容”的原则落实到代码和文档。

任务：

- 检查所有入口脚本，避免默认注入大段证据文本。
- 检查 planner 和 executor，确保通过工具拿证据。
- 检查写回逻辑，确保新观察自动进入图谱。

完成标准：

- 任意问题开始时，模型拿到的输入中不包含预先整理好的完整视频内容。

### Phase B：强化同视频多题记忆复用

目标：

- 同一视频的一组问题共享长期记忆和 session。

任务：

- 保持同视频 session 常驻。
- 保存 per-video session trace。
- 增加记忆复用统计。
- 区分“命中旧记忆”和“重新访问原始数据”。

完成标准：

- 多题运行时能看到复用次数和减少的重复访问次数。

### Phase C：把原始数据访问彻底工具化

目标：

- 让帧提取、区域定位、OCR、音频峰值、候选补帧全部通过标准工具完成。

任务：

- 完善视频访问工具。
- 完善区域和 bbox 工具。
- 完善 OCR 和音频工具。
- 统一 artifact 保存路径。

完成标准：

- 任意关键证据都能回到原始帧或区域图。

### Phase D：形成面向厨房流程的长期记忆

目标：

- 不只是记一题证据，而是形成视频级的 cooking process memory。

任务：

- 写入 ingredient timeline。
- 写入 recipe timeline。
- 写入 state change。
- 写入 location relation。
- 写入 OCR/numeric readings。

完成标准：

- 视频处理若干题后，图谱中能恢复主要 cooking timeline。

### Phase E：做真正的 agent 评测

目标：

- 不只比较准确率，还比较 agent 性。

指标：

- 准确率。
- 平均工具调用数。
- 原始数据回查率。
- 记忆复用率。
- 每题新增观察数。
- 每题重复访问减少量。
- 成本和时延。

完成标准：

- 能证明系统不是“换皮问答器”，而是具有真实检索和复用能力的 agent。

## 12.11 研究意义

这条路线的研究意义不在于“让模型再多看几张图”，而在于：

1. 在成本受限条件下，把长视频理解拆成可规划、可检索、可复用的问题求解过程。
2. 研究外部长期记忆是否能减少重复多模态观察。
3. 研究结构化记忆与原始证据回查如何结合，而不是互相替代。
4. 研究 agent 在厨房过程理解中是否比单次 VLM QA 更具可解释性、可扩展性和应用价值。

---

## 12.12 应用价值

如果这条路线做成，应用价值主要体现在：

1. 厨房过程复盘：知道什么时候做了什么，而不是只会选题。
2. 营养与配料追踪：能回查加料时刻、称重读数和状态变化。
3. 长视频问答：能针对视频内容自主找证据，而不是依赖人工整理摘要。
4. 多轮交互：用户问多个问题时，系统能累积记忆而不是重复花钱看同一段视频。
5. 可核查性：答案可以回到具体帧、具体区域、具体 OCR、具体时间段。

---

## 12.13 后续实施纪律

后续任何新增模块都应满足下面三条：

1. 不能靠增加 prompt 中的整理内容来“提分”。
2. 不能把图谱做成隐藏答案库。
3. 不能失去对原始视频、原始帧、原始区域和原始音频的可检索能力。

如果某个优化只是在“喂更多整理信息”，它不属于本项目主线。
