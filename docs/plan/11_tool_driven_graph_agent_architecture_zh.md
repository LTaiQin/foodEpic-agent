# 11 工具驱动图谱记忆 Agent 架构计划

## 11.1 文档目标

这份文档定义后续 `foodEpic-agent` 的主线形态，不再把系统做成“多模态大模型问答器”，而是做成一个会主动思考、主动调用工具、主动回查原始证据的真实 agent。

目标系统必须满足三点：

- 大模型不直接接收整理好的视频内容。
- 图谱只作为外部长期记忆层，不替代原始证据。
- 模型始终保留对原始视频、原始帧、局部区域、OCR、音频和跨视频参考图的检索能力。

---

## 11.2 核心原则

### 11.2.1 不主动把内容塞给模型

默认情况下，不允许直接输入给大模型：

- 全量视频摘要。
- 全量关键帧描述。
- 人工筛好的证据包。
- 人工整理好的流程文本。
- 某个题目的定制答案提示。

默认只允许输入给大模型：

- 当前问题。
- 当前选项。
- 工具列表与工具 schema。
- 当前工作记忆。
- 上一步工具返回结果。

也就是说，模型看到的是“环境”，不是“现成答案”。

### 11.2.2 图谱是记忆层，不是终点

图谱里保存的是可检索、可写回、可复用的结构化观察，而不是最终答案。

任何图谱证据都必须能反查到原始来源：

- 原始视频。
- 时间点或时间段。
- 抽取帧。
- 画框图。
- 局部区域图。
- OCR 结果图。
- 音频片段。

如果一个图谱节点不能回到原始证据，它就不能被视为强证据。

### 11.2.3 图谱中只存评测允许的信息

允许进入图谱：

- 视频与音频本身。
- 自动抽帧结果。
- 自动区域图。
- 自动 OCR 结果。
- 自动提取的对象、动作、状态、位置、时间事件。
- agent 在执行过程中写回的观察结论。

禁止进入图谱：

- benchmark 标准答案。
- 标注真值本身。
- 测试时理论上不可见的信息。

---

## 11.3 最终系统形态

最终系统流程必须是：

`问题 -> planner -> 工具调用 -> 工作记忆更新 -> 判断是否缺证据 -> 再检索/再回查 -> 最终回答`

而不是：

`问题 -> 大 prompt -> 模型直接回答`

系统至少由五层组成：

1. `Memory Layer`
2. `Tool Layer`
3. `Planner Layer`
4. `Execution Layer`
5. `Evaluation Layer`

---

## 11.4 各层职责

### 11.4.1 Memory Layer

职责：

- 保存跨步骤、跨问题、跨时间段的长期记忆。
- 保存节点与边，不保存题目答案。
- 支持回查原始数据路径。

建议节点类型：

- `video`
- `segment`
- `frame`
- `region`
- `object_track`
- `ingredient_event`
- `state_change`
- `recipe_step`
- `timeline_event`
- `ocr_reading`
- `audio_event`
- `cross_video_reference`
- `agent_observation`

建议边类型：

- `contains`
- `before`
- `after`
- `supports`
- `derived_from`
- `same_object`
- `same_ingredient`
- `same_step`
- `refers_to`
- `co_occurs`

每个节点至少包含：

- `node_id`
- `node_type`
- `video_id`
- `start_time`
- `end_time`
- `label`
- `attributes`
- `evidence_paths`
- `source_tool`
- `confidence`

### 11.4.2 Tool Layer

职责：

- 所有证据访问都必须经过工具。
- 工具返回结构化结果，而不是随意长文本。
- 工具必须能被 planner 明确选择。

工具分四类：

1. 图谱检索工具
2. 原始视频访问工具
3. 感知解析工具
4. 图谱写回工具

### 11.4.3 Planner Layer

职责：

- 识别题型。
- 判断当前缺什么证据。
- 选择最合适的下一步工具。
- 判断是否需要继续回查原始数据。

planner 不负责直接“看完所有内容后给答案”，而只负责做决策。

### 11.4.4 Execution Layer

职责：

- 实际执行工具。
- 维护工作记忆。
- 控制最大步数、失败重试、暂停恢复。
- 记录 trace、artifact 和中间证据。

### 11.4.5 Evaluation Layer

职责：

- 跑多任务评测。
- 保存每题 trace。
- 保存失败原因。
- 支持断点续跑。
- 统计准确率、成本、工具使用分布与证据质量。

---

## 11.5 模型在系统中的职责边界

模型只负责：

1. 判断当前问题需要什么证据。
2. 选择工具。
3. 判断证据是否足够。
4. 基于证据输出答案。

模型不负责：

1. 不经工具直接做长视频理解。
2. 靠大 prompt 读整理好的摘要答题。
3. 把图谱当作最终答案库。

---

## 11.6 工作记忆设计

建议统一维护如下状态：

```json
{
  "video_id": "",
  "task_family": "",
  "question": "",
  "choices": [],
  "hypotheses": [],
  "evidence": [],
  "visited_times": [],
  "visited_nodes": [],
  "artifacts": [],
  "open_questions": [],
  "confidence": 0.0
}
```

执行约束：

- 每一步都只能基于当前工作记忆和工具结果继续。
- 不能在 planner 阶段绕过工具直接大量推断。
- 工作记忆只保留当前问题相关的核心证据，不无限膨胀。

---

## 11.7 工具体系设计

### 11.7.1 图谱检索工具

必须支持：

- `query_time`
- `query_object`
- `query_event`
- `get_neighbors`
- `query_state`
- `query_location`
- `query_region`
- `query_ocr`

这些工具解决的是“先低成本拿已有结构化记忆”。

### 11.7.2 原始视频访问工具

必须支持：

- `extract_frame_at_time`
- `extract_frames_for_range`
- `sample_sparse_frames`
- `render_bbox_overlay`
- `render_bbox_sequence`
- `extract_region_with_context`
- `extract_input_reference_frames`

这些工具解决的是“图谱不够时，重新看原始证据”。

### 11.7.3 感知解析工具

必须支持：

- `inspect_visual_evidence`
- `run_ocr_on_image`
- `run_ocr_on_region`
- `detect_audio_peaks`
- `summarize_audio_segment`
- `identify_image_ingredients`
- `infer_gaze_target_with_context`
- `infer_viewpoint_choice`
- `infer_visual_mcq`

这里的关键是：

- 不是把全部帧描述预先喂给模型。
- 而是需要时，agent 自己调用感知工具去看指定帧、指定区域、指定时间段。

### 11.7.4 图谱写回工具

必须支持：

- `write_observation`
- `write_frame_observation`
- `write_region_observation`
- `write_ocr_reading`
- `write_audio_event`
- `write_timeline_summary`
- `write_state_change`

写回时必须带：

- 时间范围。
- 来源工具。
- 证据路径。
- 置信度。

这样后续问题才能复用已经观察过的证据，而不是每题都从零开始。

---

## 11.8 视频处理策略

### 11.8.1 不能依赖密集全帧

由于成本限制，不能直接把整段视频按高频率抽帧后全部送模型。

需要采用分层抽帧：

1. 稀疏全局扫描。
2. 关键时间点局部补帧。
3. 候选时间窗口二次细化。
4. 区域画框与局部放大。

### 11.8.2 关键时间点来源

关键时间点不只来自题目本身，还应来自：

- 题目给定时间。
- `inputs_json` 的参考图或参考视频时间。
- 音频峰值。
- 图谱中已知事件。
- 动作/状态变化节点。

### 11.8.3 抽帧原则

- 不抽连续相邻帧。
- 默认使用稀疏采样。
- 如果某时间点附近证据不足，再在该窗口内二次补帧。
- 对定位题默认画框，不默认直接裁剪。
- 只有 OCR、小目标读数、称重显示这类任务才优先局部放大。

---

## 11.9 知识图谱与原始数据的关系

知识图谱不应替代原始视频，而应承担三个作用：

1. 作为外部长期记忆。
2. 作为中间观察结果的写回容器。
3. 作为下一步工具检索的入口。

最终结构应是：

- 图谱存“我知道哪里可能有证据”。
- 原始视频存“真正可验证的证据”。
- 工具负责“从图谱跳回原始数据，再把新观察写回图谱”。

---

## 11.10 针对 HD-EPIC VQA 的实际执行策略

### 11.10.1 时间已给定的问题

这类题优先：

1. 检索给定时间附近的图谱节点。
2. 若不足，再在该时间点前后抽帧。
3. 若涉及定位，再画框。
4. 若涉及文本读数，再 OCR。

### 11.10.2 时间未完全给定的问题

这类题优先：

1. 先根据问题解析对象、动作、目标状态。
2. 在图谱中搜候选事件。
3. 对候选事件分别抽少量帧验证。
4. 比较候选证据后再回答。

### 11.10.3 跨视频参考图问题

这类题必须支持：

1. 从 `inputs_json` 中提取引用图。
2. 保留引用图与当前视频的关联。
3. 先识别参考图对象，再在当前视频里回查对应交互或状态。

### 11.10.4 重量、OCR、营养题

这类题不能只靠语言猜。

应优先链路：

1. 题目解析食材名或营养目标。
2. 检索 ingredient/state 节点。
3. 若存在称重显示器或包装信息，回看局部区域并 OCR。
4. 把结构化数值写回图谱。
5. 再做比较与回答。

---

## 11.11 研究价值

这个方向真正有研究价值的地方，不是“让模型回答厨房问题”，而是：

1. 在长视频、受成本约束的条件下，如何让 agent 自主选择最小必要证据。
2. 如何用图谱作为外部长期记忆，而不是一次性 prompt。
3. 如何让 agent 在图谱记忆和原始视频之间来回跳转。
4. 如何让证据可追溯、可复查、可写回、可复用。
5. 如何在多任务厨房场景中体现 agent 相比普通大模型的优势。

---

## 11.12 应用价值

最终系统的应用价值应体现在：

- 自动记录做饭过程。
- 自动形成可检索的烹饪时间线。
- 回答“什么时候加了什么、加了多少、为什么这样做”的过程问题。
- 回答“这个东西在哪里、何时拿起、何时放下、是否还在原处”的物体问题。
- 检测漏步骤、重复步骤、顺序异常和称量异常。
- 支持厨房教学复盘、健康饮食记录和家庭辅助记忆。

---

## 11.13 分阶段实施路线

### 阶段 A：固化工具驱动约束

目标：

- 清理所有“主动把整理内容塞给模型”的旧入口。
- 统一 planner 只看工具、问题和工作记忆。

完成标准：

- planner/executor 不再直接消费大块摘要文本。
- 所有证据都来自工具返回。

### 阶段 B：补齐图谱检索与写回闭环

目标：

- 扩展 `query_state`、`query_location`、`query_ocr`、`query_region`。
- 扩展写回工具，使中间观察长期可复用。

完成标准：

- agent 能先检索图谱，再把新观察写回，再继续检索。

### 阶段 C：补齐原始视频回查能力

目标：

- 增强稀疏抽帧、二次补帧、bbox 序列可视化、局部上下文放大、OCR 和音频峰值检测。

完成标准：

- 对任何需要原始证据的问题，agent 都能自行回查，而不是依赖预先准备好的图片描述。

### 阶段 D：单视频完整闭环验证

目标：

- 选择代表视频。
- 为该视频建立图谱。
- 让 agent 自主调用工具完成一组多类型 VQA。

完成标准：

- 每道题保留完整 trace、artifact、工作记忆和图谱写回记录。

### 阶段 E：多视频多任务评测

目标：

- 扩大到多视频、多任务族评测。
- 保持断点续跑、错误暂停、失败重试。

完成标准：

- 输出准确率、任务族分数、失败分类、成本、工具使用频次与证据命中率。

### 阶段 F：形成论文级实验对比

目标：

- 与 `TextOnly`、`DirectEvidence`、`FoodState`、`LightAgent baseline`、`GraphAgent` 做公平对比。

完成标准：

- 明确证明工具驱动图谱记忆 agent 在准确率、可解释性、证据质量或成本效率上具备优势。

---

## 11.14 评测指标建议

除了最终准确率，还必须统计：

- `Accuracy`
- `Task-family accuracy`
- `Evidence sufficiency rate`
- `Raw-video revisit rate`
- `Graph write-back reuse rate`
- `Average tool steps`
- `Average cost per sample`
- `Failure type distribution`

如果只看最终选择题准确率，很难体现 agent 的真实优势。

---

## 11.15 后续实现约束

从本文件开始，后续实现必须遵守以下规则：

- 不再增加“先整理内容再喂模型”的新路径。
- 新能力优先实现为工具，而不是更长的提示词。
- 图谱设计必须保留原始证据路径。
- 任何中间观察都尽量可写回、可复用、可再次检索。
- 评测脚本必须保留可恢复、可审计、可复查的运行记录。

这份文档不是建议，而是后续 `foodEpic-agent` 的实现边界。
