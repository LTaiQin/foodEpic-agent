# 评测结果与下一步计划

## 20 样本评测结果

| 方法 | 正确率 | 平均耗时 | 总 tokens |
|---|---|---|---|
| **MiMo + Video (直答)** | **8/20 = 40%** | 13s | 56,324 |
| Agent (observation-centric) | 7/20 = 35% | 61s | 195,607 |
| MiMo Text-only (无视频) | 3/20 = 15% | 15s | 30,781 |

## 关键发现

1. **视频是核心**: 40% vs 15%，视频输入比纯文本高 2.7x
2. **Agent 略低于直答**: 35% vs 40%，但 agent 赢了 3 题 video baseline 错的题（互补）
3. **所有 20 题都以 `budget_exhausted` 结束** — 这是根本问题

## 根本问题：max_steps 不应该是停止条件

当前 `max_steps=6` 是硬循环上限（executor 的 `for step_index in range(max_steps)`）。agent 还在搜证阶段就被切断，被迫猜答案。

**正确设计**：
- `max_steps` 设大（15-20）作为**安全网**，防止无限循环
- **真正的停止条件**是模型自己判断：
  - verifier 说 `sufficient=True` → 停止
  - planner 说 `finish` 且 verifier 确认 → 停止
  - 所有 gap 都已尝试但无法闭合 → 保守停止
- `search_budget` 中的 `max_new_frames`（24）和 `max_long_horizon_expansions`（2）才是真正的资源约束

## 下一步实现计划

### 步骤 1：调整 max_steps 为安全网
- eval 脚本的 `--max-steps` 默认改为 15
- executor 的循环上限不变，但不应成为主要停止机制
- `search_budget` 的其他约束（frames、expansions）继续生效

### 步骤 2：重新跑 20 样本 eval
- 用 `--max-steps 15` 重跑
- 观察是否有题以 `finish_confident` 或 `needs_more_evidence` 结束（而非全部 budget_exhausted）
- 对比正确率变化

### 步骤 3：集成 inspect_video_evidence
- 当前 agent 用的是抽帧看图，应该用视频理解
- 在 planner 的 timeline review 路径中已部分集成
- 需要在 eval 中验证效果

### 步骤 4：对比分析
- Agent (max_steps=15, 抽帧) vs Agent (max_steps=15, 视频) vs Video baseline
- 统计每题的 stop_reason 分布
